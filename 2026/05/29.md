## ⭐️Cloud NAT と Private Google Access (PGA) の関係

- Cloud NAT は `*.googleapis.com` 宛トラフィックを NAT 翻訳しない。"PGA が有効ならそちらに譲る"仕様

> "Traffic sent to Google APIs and services are routed through Private Google Access even if the VM instance initiating the connections uses Public NAT." — [Cloud NAT overview](https://cloud.google.com/nat/docs/overview)

- PGA 経由のトラフィックの送信元 IP は VM の内部 IP (RFC1918) として Google API 側に見える

## Google API キーの IP 制限の制約

- API キーの "サーバー IP アドレス制限" は **public IP のみ** 受付。内部 IP (RFC1918) や `localhost` は登録不可

> "Internal IP addresses and `localhost` aren't supported." — [API key restrictions](https://cloud.google.com/docs/authentication/api-keys)

- そのため Cloud Run (外部 IP なし) + PGA 構成からの Google API 呼び出しは、API キー IP 制限とは原理的に共存できない

## OTel Collector 経由で Claude Code のテレメトリを Google Cloud に送ったら一部ログが破棄された件

### 事象
- ローカルの Claude Code → GLB → Cloud Run 上の OTel Collector → telemetry.googleapis.com の構成でテレメトリ送信を試した。
- Collector のログに以下のエラーが継続的に出て、ログが破棄されていた。
  - `code = InvalidArgument`
  - `desc = Failed to parse attribute value to int64: ... string_value: "connected"` / `"failed"`

### 原因
- Google Cloud Telemetry API は、OTLP ログの属性キー `status` を LogEntry.httpRequest.status（数値）にマップする予約キーとして扱う。
- Claude Code の `claude_code.mcp_server_connection` イベントは仕様上 `status` 属性に `"connected"` / `"failed"` / `"disconnected"` の文字列を入れる。
- 両者の命名が衝突し、GCP 側で数値パースに失敗 → 該当リクエストのログが丸ごと破棄されていた。

### 対処
- Collector の logs パイプラインに transform プロセッサを追加し、エクスポート前に `status` 属性を `mcp.connection.status` にリネームして GCP のマッピング対象から外す。
- 値そのものは文字列として保持されるため、Cloud Logging でも引き続き参照可能。

### 学び・今後の注意
- OTel の attribute キーは自由に命名できるが、GCP Telemetry API 側に予約キー（status, latency, http.response.status_code, code.function.number など）が存在し、型解釈の衝突が起きうる。
- OTel ドキュメントの「承認済みのタイプ: 文字列、整数」は「任意の文字列 OR 整数」ではなく「数字を表す文字列 OR 整数」の意味だった。
- 計装側（Claude Code など）を直せない場合は、中継する Collector 側で吸収するのが現実解。
- 新たに `Failed to parse attribute value to int64` を観測したら、衝突キーを特定して同じ要領でリネームルールを追加していく。

### 参考
- GCP マッピング仕様: https://cloud.google.com/stackdriver/docs/reference/telemetry/otlp-log-record-to-log-entry
- Claude Code テレメトリ仕様: https://docs.claude.com/en/docs/claude-code/monitoring-usage

## xattr

### 概要
- macOSには Gatekeeper というセキュリティ機能があり、インターネットからダウンロードしたアプリに「隔離フラグ（quarantine）」を自動で付ける
- このフラグが付いたアプリをAppleが検証できなかった場合、「壊れている」として起動をブロックする
- xattr は extended attributes（拡張属性） を操作するコマンド
- macOSのファイルには通常のデータ以外に、メタ情報（拡張属性）を付けられる
- Gatekeeperの隔離フラグもその一つ

```shell
xattr -cr /Applications/Claude.app
```

- `-c`
  - そのファイル/ディレクトリの全拡張属性を削除
- `-r`
  - ディレクトリを再帰的に処理（.appの中身も含めて）
