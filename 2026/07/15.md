## ⭐️tini

https://github.com/krallin/tini

- コンテナ用の`超軽量init`プロセス
- 一言でいうと「コンテナの中に本物の`PID 1`（init）を立ててあげるツール」

### 前提: なぜコンテナは`PID 1`問題を抱えるのか

- 通常のLinuxマシンでは`PID 1`は`init`（`systemd`など）で、以下を専門に担当している
    - シグナルの正しい転送・処理
    - 親を失った孤児プロセスを引き取り、終了時に`wait()`で回収する
- コンテナではこの`init`を省略し、アプリ自身が`PID 1`になるケースが多い
    - しかしアプリは「自分がinitになる」ことを想定して作られていないため、上記の役割を果たせない

### 役割（何を代行するか）

- **シグナルの転送**: `docker stop`（`SIGTERM`）などを受け取り、子プロセス（実際のアプリ）へ正しく転送する
- **ゾンビプロセスの回収（reap）**: 孤児化した子プロセスを引き取り、`wait()`して片付ける

### よくある使い方

- Dockerに`tini`が組み込み済みなので、`--init`フラグを付けるだけでも良い
    ```shell
    docker run --init myimage
    ```
    - このフラグを付けると、Dockerエンジンが`docker-init`（静的リンクされた`tini`相当）を**自動的に注入**してPID 1にする
    - イメージ側（`myimage`）は何も変更しなくてよい。Docker側が「`docker-init` → アプリ」という親子関係を作ってくれる
        ```
        docker run --init myimage
                │
                ▼
          Dockerが docker-init(tini相当) を注入し、PID 1として起動
                │  fork + exec
                ▼
          docker-init(PID 1) ──▶ your-app(子プロセス)
                │
                └─ SIGTERM転送・ゾンビ回収を代行
        ```
    - ただしこれは`docker run`のフラグなので、起動コマンドを自前で指定できないCloud Run等では使えない（→ 後述の`ENTRYPOINT`方式が必要）
- Dockerfileで明示的に`ENTRYPOINT`に指定する方法もよく使われる（アプリは`CMD`側に書く）
    ```dockerfile
    ENTRYPOINT ["/tini", "--"]
    CMD ["your-app"]
    ```

### Cloud Runなど`docker run`できない環境の場合

- Cloud RunやKubernetesなどは起動コマンドを自前で指定できず、`--init`フラグは使えない
- その場合は`ENTRYPOINT`方式一択。イメージに`tini`を焼き込んでおけば、実行環境側の起動方法によらず常にPID 1として動く
    ```dockerfile
    RUN apt-get update && apt-get install -y tini && rm -rf /var/lib/apt/lists/*
    ENTRYPOINT ["/usr/bin/tini", "--"]
    CMD ["your-app"]
    ```

#### このコマンドの意味

- `ENTRYPOINT`と`CMD`はexec形式（配列表記）だと連結され、実際には `/usr/bin/tini -- your-app` というコマンドラインが実行される
    - `/usr/bin/tini`: `tini`実行ファイルのフルパス。`tini`だけ（フルパスなし）でも`PATH`が通っていれば動くが、`ENTRYPOINT`は生死を握る最重要プロセスなので`PATH`依存を避けてフルパス指定するのが公式推奨
    - `--`: 「オプションはここまで」を示す区切り。以降の`your-app`側の引数を`tini`自身のオプションと誤解釈させないための安全策
    - `your-app`: `tini`が実際に起動する対象プロセス（`CMD`側）

#### Cloud Runでの内部動作

```
① Cloud Runがインスタンスを起動
        │
        ▼
② コンテナ起動: /usr/bin/tini -- your-app が実行される
        │
        ▼
③ tini が PID 1 として起動
        │  fork + exec
        ▼
④ your-app が子プロセスとして起動（リクエスト処理を開始）
        │
        ▼
⑤ Cloud Runがインスタンスを停止する時
     Cloud Run ──SIGTERM──▶ tini(PID 1) ──転送──▶ your-app
                                                        │ graceful shutdown
                                                        ▼
                                                       exit
                                                        │
                                          tini が wait() で回収
                                                        ▼
                                        tini も同じ終了コードでexit
                                                        ▼
                                                 コンテナ終了 ✅
```

- `tini`はアプリをexec置換するのではなく`fork`で子として起動し、自分はPID 1に居座り続ける
- これにより`SIGTERM`を正しく中継でき、`your-app`が生成する孫プロセスのゾンビ化も防げる
