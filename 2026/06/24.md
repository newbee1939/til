## Microsoft Entra

- マイクロソフトが提供する、クラウドベースのアイデンティティおよびアクセス管理（IAM）サービスのファミリー。旧称は「`Azure AD（Active Directory）`」
- ユーザーの認証、認可、シングルサインオン（SSO）、多要素認証（MFA）、およびデバイス管理を一元的に行う
- 「条件付きアクセス」機能により、ユーザーの場所、デバイスのセキュリティ状態、リスクレベルなどに応じて柔軟にアクセス制御ができる点が強力

### Q. 「クラウドベースのIAM」とは？

IAM（Identity and Access Management）は「**誰が・何に・どうアクセスできるか**」を一元管理する仕組み。

| 概念 | 意味 | 具体例 |
|------|------|--------|
| **Identity**（アイデンティティ） | 誰であるかの識別 | ユーザーアカウント、グループ、サービスアカウント |
| **Access Management**（アクセス管理） | 何にアクセスできるか | AアプリはOK・Bファイルは閲覧のみ、など |

「クラウドベース」とは、オンプレミス（社内サーバー）ではなくMicrosoftのクラウド上で動作するという意味。社内にADサーバーを立てなくても、インターネット経由でID管理ができる。

### Q. 「デバイス管理」とは？

社員が業務で使うMac・Windows PC・iPhone・Androidスマートフォンなどのデバイスを、組織として把握・管理すること。一般に**MDM（Mobile Device Management）**とも呼ばれる。元々スマートフォン管理が主目的だったが、現在はPCも含む総称として使われる。

- **デバイス登録:** 会社が管理するデバイスをEntra IDに登録し、「このMacは社用端末」と把握する
- **コンプライアンスチェック:** 「OSが最新か」「ディスク暗号化されているか」などの条件を自動確認する
- **条件付きアクセスとの連携:** 「コンプライアンス未準拠のデバイスからのアクセスは禁止」というポリシーを適用できる

> MacはMicrosoft Intune（EntraファミリーのMDMサービス）や **Jamf**（Mac特化のMDM）と連携して管理することが多い。

## Microsoft EntraとGoogle Workspaceとの連携

- **フェデレーション（ID連携）:** Google Workspaceの認証元（IdP）としてMicrosoft Entra IDを設定し、`SAML 2.0`などのプロトコルを用いてシングルサインオン（SSO）を実現する。これにより、ユーザーがGoogleにログインしようとすると、自動的にMicrosoftのサインイン画面にリダイレクトされる

### SSO（シングルサインオン）の仕組み：SAML 2.0を使った場合

まず用語の整理：

| 役割 | 意味 | 今回の例 |
|------|------|---------|
| **IdP**（Identity Provider） | 「この人は誰か」を証明する側 | Microsoft Entra ID |
| **SP**（Service Provider） | サービスを提供する側（アクセスされる側） | Google Workspace |

```
ユーザー        Google Workspace (SP)       Microsoft Entra ID (IdP)
   |                    |                              |
   |--① Googleにアクセス-->|                              |
   |                    |--② SAMLリクエスト送信--------->|
   |<-----③ Entraログイン画面へリダイレクト---------------|
   |                    |                              |
   |--④ ID/PW + MFA入力------------------------------>|
   |                    |              ⑤ 認証成功・アサーション生成
   |                    |<--⑥ SAMLアサーション（XML）送信--|
   |                    |--⑦ アサーション検証            |
   |<--⑧ アクセス許可---|                              |
```

**SAMLアサーション**とは「このユーザーは認証済みで、このメールアドレス・属性を持っています」という情報をXML形式で記述した署名付き文書。GoogleはこのXMLを受け取り、Entra側の電子署名を検証してログインを許可する。

### OAuth / OIDC との違い（参考）

同じ「ID連携」でもプロトコルによって用途が異なる：

| プロトコル | 主な用途 | 発行するもの | データ形式 |
|-----------|---------|------------|---------|
| **OAuth 2.0** | **認可**（アクセス権の委譲） | アクセストークン | JSON |
| **OIDC**（OpenID Connect） | **認証** + 認可 | IDトークン（JWT）+ アクセストークン | JSON |
| **SAML 2.0** | **認証** + SSO | SAMLアサーション | XML |

> OIDCはOAuth 2.0の拡張版。OAuthは「BアプリがAさんのGoogleドライブにアクセスする権限を委譲する」（認可）のための仕組みで、「誰がログインしているか（認証）」は保証しない。OIDCはその上にIDトークン（JWT）を追加して認証もできるようにしたもの。エンタープライズSSO（EntraとGoogle Workspaceの連携など）ではSAML 2.0が多く使われる傾向にある。

## TOTP

- `Time-based One-Time Password`
- 現在時刻とあらかじめ共有した秘密鍵を基に、一度だけ使用可能な使い捨てパスワード（ワンタイムパスワード）を生成する仕組み
- 一般的に30秒などの一定時間ごとに新しいパスワードが生成され、一度使用された、または有効期限が切れたパスワードは二度と使用できないため、フィッシングや盗聴に対して強い耐性を持つ

## Microsoft Entra TOTP

- Microsoft Entra IDにおける`多要素認証（MFA）`の認証手段の一つ。
- ユーザーは「Microsoft Authenticator」などの認証アプリをスマートフォンにインストールし、表示される6桁〜8桁の動的な数字を入力してログインを完了させる
- Google WorkspaceとのSSO連携（フェデレーション）を組んでいる場合、Googleへのログイン時にもこのEntra ID側のTOTP（MFA）を強制することが可能になり、組織全体のセキュリティレベルを均一に高めることができる

### TOTPコードの生成原理

```
  ┌──────────────────┐     ┌──────────────────────┐
  │  秘密鍵（共有済）  │     │  現在時刻（30秒単位） │
  │  （初回セットアップ│  +  │  例: 1750000020      │
  │   時にQRコード等で │     │  ← 30秒ごとに変わる  │
  │   アプリと共有）   │     └──────────────────────┘
  └──────────────────┘               │
              │                      │
              └──────────┬───────────┘
                         ▼
                  HMAC-SHA1 演算
                         │
                         ▼
                ┌─────────────────┐
                │  6桁コード生成   │  例: 8 4 7 3 9 2
                │  30秒後には別の  │  ← 使い捨て・時限式
                │  コードになる    │
                └─────────────────┘
```

ポイントは**秘密鍵と時刻の両方が揃わないと同じコードを生成できない**こと。攻撃者がコードを盗み見ても30秒で無効になるため、フィッシングや盗聴に強い。

### Entra MFA（TOTP）の認証フロー

```
ユーザー                  Microsoft Entra ID               認証アプリ
   |                            |                        (Authenticator等)
   | ① ID/パスワード入力（第1要素）|                              |
   |--------------------------->|                              |
   |          ② 第1要素OK・MFA要求|                              |
   |<---------------------------|                              |
   |                            |                              |
   | ③ アプリを開いて6桁コードを確認 ────────────────────────>|
   |    （秘密鍵 + 現在時刻で生成）                            |
   |<──────────────────────────── ④ コード表示（例: 847392）  |
   |                            |                              |
   | ⑤ 6桁コード入力（第2要素）   |                              |
   |--------------------------->|                              |
   |                ⑥ コード検証 |                              |
   |         （Entra側も同じ秘密鍵|                              |
   |          +現在時刻でコードを  |                              |
   |          生成し、一致確認）   |                              |
   |             ⑦ 認証成功      |                              |
   |<---------------------------|                              |
```

> **なぜEntra側でも同じコードを生成できるの？**
> セットアップ時に秘密鍵をEntra IDとユーザーのアプリの両方に共有している。そのため「同じ秘密鍵 × 同じ時刻」から同じコードを独立に生成でき、ネットワーク越しにコードを送受信することなく検証できる（**共有秘密鍵方式**）。

### Q. MFA強制はEntraの管理画面で設定できるの？

**はい、設定できます。** 主な設定箇所は以下の2つ：

**① 認証方法ポリシー（Authentication Methods Policy）**
- Entra管理画面（entra.microsoft.com）→「保護」→「認証方法」
- TOTPを認証手段として有効化し、ユーザー/グループに割り当てる

**② 条件付きアクセスポリシー（Conditional Access Policy）**
- 「保護」→「条件付きアクセス」→「新しいポリシー」から設定
- 「誰が・どのアプリに・どんな条件でアクセスする場合に・何を要求するか」を定義できる

## otplib × Playwright でGoogleアカウントのTOTP認証を突破してログインする

> **前提（重要）:** これは自分が管理するアカウントのE2E（End-to-End）テスト自動化を想定した手法。第三者のアカウントへの不正ログインは不正アクセス禁止法に抵触するためNG。CI/CDでGoogleログインを伴う画面をテストしたい、といった正当なユースケースで使う。

### 全体像：なぜ「自動化」が可能なのか

通常、TOTPの6桁コードは**スマホの認証アプリ**が生成する。これを人間が目で見て手入力するのが普通のフロー。

しかしTOTPコードは「**秘密鍵 + 現在時刻**」だけで計算できる（前述の生成原理）。つまり**秘密鍵さえ手元にあれば、スマホがなくてもプログラムが同じ6桁コードを生成できる**。この「秘密鍵からコードを計算する」役割を担うのが [`otplib`](https://github.com/yeukfei02/otplib)（Node.jsのTOTPライブラリ）。

```
  ┌─────────────────────────────────────────────────────────┐
  │  普段の人間のログイン                                     │
  │                                                         │
  │   スマホアプリ ──[6桁コードを目視]──> 人間 ──[手入力]──> Google │
  └─────────────────────────────────────────────────────────┘
                          ↓ これを丸ごとプログラムに置き換える
  ┌─────────────────────────────────────────────────────────┐
  │  自動化したログイン                                      │
  │                                                         │
  │   otplib ──[6桁コードを計算]──> Playwright ──[自動入力]──> Google │
  │     ▲                                                   │
  │   秘密鍵（環境変数などで安全に保管）                       │
  └─────────────────────────────────────────────────────────┘
```

- **otplib** … 秘密鍵と現在時刻から6桁コードを生成する（スマホ認証アプリと同じ計算をするライブラリ）
- **Playwright** … ブラウザを自動操作するツール。メール・パスワード・6桁コードの入力やボタンクリックを人間の代わりに実行する

### 事前準備：秘密鍵（TOTP Secret）の入手

Googleアカウントで2段階認証の「認証システムアプリ」を登録する際、通常はQRコードを表示してスマホで読み取る。このとき画面の「**QRコードをスキャンできない場合**」というリンクを押すと、**Base32形式の文字列（秘密鍵）**が表示される。

```
例: JBSW Y3DP EHPK 3PXP
```

この文字列こそがotplibに渡す秘密鍵。**一度きりしか表示されない**ので控えておく。漏洩すると誰でもコードを生成できてしまうため、Gitにコミットせず環境変数やシークレットマネージャーで管理する。

> QRコードの実体は `otpauth://totp/...?secret=JBSWY3DPEHPK3PXP&...` というURL。この `secret=` の値がBase32の秘密鍵そのもの。

### コード例

```ts
import { chromium } from 'playwright';
import { authenticator } from 'otplib';

const EMAIL = process.env.GOOGLE_EMAIL!;
const PASSWORD = process.env.GOOGLE_PASSWORD!;
const TOTP_SECRET = process.env.GOOGLE_TOTP_SECRET!; // Base32の秘密鍵

const browser = await chromium.launch();
const page = await browser.newPage();

// ① ログイン画面へ
await page.goto('https://accounts.google.com/signin');

// ② メールアドレス入力 → 次へ
await page.fill('input[type="email"]', EMAIL);
await page.click('#identifierNext');

// ③ パスワード入力 → 次へ（第1要素）
await page.fill('input[type="password"]', PASSWORD);
await page.click('#passwordNext');

// ④ TOTPコードを「その場で」生成（第2要素）
//    authenticator.generate() が 秘密鍵 + 現在時刻 から6桁を計算する
const token = authenticator.generate(TOTP_SECRET); // 例: "847392"

// ⑤ 6桁コードを入力して認証完了
await page.fill('input[type="tel"]', token);
await page.click('#totpNext');

await page.waitForURL('https://myaccount.google.com/**');
console.log('ログイン成功');
```
