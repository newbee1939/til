## PostgreSQL DB オブジェクト所有者

- PostgreSQLオブジェクトとは、データベース内に作成・格納されるすべての構成要素（テーブル、ビュー、インデックス、関数など）の総称
- PostgreSQLではデータベースをはじめ、テーブルや関数などの各種「オブジェクト」が作成されると、必ずその所有者（オーナー）が割り当てられる

### 誰が所有者になるのか？

- 通常、そのオブジェクト（データベース、テーブルなど）を作成する文（CREATE DATABASE や CREATE TABLE など）を実行したロール（ユーザ）が自動的に所有者となる
- あとで所有者の変更は可能

## Claude Managed Agents: memory

https://platform.claude.com/docs/en/managed-agents/memory

- Managed Agentsのセッションはデフォルトで毎回まっさらなコンテキストから始まり、セッション終了で状態は消える。memory store を使うとセッションをまたいで情報（ユーザーの好み、プロジェクト規約、過去のミス、ドメイン知識）を引き継げる
- memory store
  - ワークスペース単位のテキストドキュメント集。セッションに紐付けると、サンドボックス内の `/mnt/memory/` 配下にディレクトリとしてマウントされる
  - エージェントは通常のファイルツールで読み書きでき、書き込みはstoreに永続化されて共有セッション間で同期される
  - 個々のmemoryは最大100kB（約25kトークン）、1 storeあたり最大2,000件。小さく焦点を絞ったファイルを多数置く構成が推奨
- 利用の流れ
  - `memory_stores.create` でstoreを作成（`name` と、エージェントに渡される `description` を指定）。事前にコンテンツをseedすることも可能
  - セッション作成時の `resources[]` でアタッチする（実行中のセッションへの追加・削除は不可、1セッション最大8 store）
  - `access` は `read_write`（デフォルト）か `read_only`。プロンプトインジェクション対策として、変更不要な参照用storeは `read_only` にするのが安全
- 管理・監査
  - API/ConsoleからmemoryをCRUD可能。更新時に `content_sha256` をpreconditionに渡せば楽観的並行制御で上書き衝突を防げる
  - 変更ごとにimmutableな memory version が作られ、監査証跡・特定時点へのロールバック・redact（機密情報の除去）ができる（バージョンは原則30日保持）
- memoryの更新は誰がやる？
  - `read_write` でアタッチすれば、エージェントが自分で読み書きする（開発者がAPIを叩かなくてよい）。ただしagent toolsetの有効化が前提で、いつ読み書きするかは `instructions` で誘導する
  - `read_only` なら書き込み不可（参照専用）。APIから手動でCRUDすることも可能
